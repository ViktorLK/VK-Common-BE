# アーキテクチャ監査レポート: Generators BuildingBlock

**対象モジュール**: `src/BuildingBlocks/Generators`
**監査日**: 2026-04-16
**監査者**: VK.Blocks Architect (AI)

---

## 📊 監査サマリー (Audit Summary)

- **総合スコア**: **82 / 100**
- **対象レイヤー判定**: Infrastructure Layer — Roslyn Source Generator / Analyzer
- **総評 (Executive Summary)**:

Generators モジュールは、VK.Blocks フレームワーク全体のコード生成基盤として、**高いアーキテクチャ的成熟度**を示している。`IIncrementalGenerator` の一貫した活用、`VKBlockGeneratorGuard` による実行ガード、`Fnv1aHash` によるメタデータ変更検出メカニズム、そしてフィーチャー駆動型フォルダ構成は、このモジュールが戦略的に設計されたことを物語る。

しかしながら、以下の領域で改善が必要である：

1. **One File, One Type 原則の違反** — `VKBlockDiagnosticsGenerator.cs` 内に `BlockInfo` レコードが同居
2. **ハッシュ計算ロジックの重複** — 中央集権化された `Fnv1aHash` ユーティリティが存在するにも拘わらず、`PermissionsCatalogGenerator` が独自のハッシュ実装を持つ
3. **生成コード内の `#region` 使用** — `PermissionsCatalogGenerator` が生成コードに `#region` ディレクティブを出力
4. **生成ヘッダーの不統一** — 一部ジェネレーターが `SourceCodeBuilder.CreateWithHeader()` を使用する一方、他は手動で `// <auto-generated/>` を出力
5. **`ProviderInfo` レコードの同一ファイル内宣言** — 複数のジェネレーターで `private sealed record` が同一クラス内に宣言されている

---

## 🚨 重大なアーキテクチャの懸念事項 (Critical Architectural Smells)

### ❌ One File, One Type 原則違反 (Rule 14)

**場所**: [`VKBlockDiagnosticsGenerator.cs`](/src/BuildingBlocks/Generators/Observability/VKBlockDiagnosticsGenerator.cs) L11-L19

`BlockInfo` レコードが `VKBlockDiagnosticsGenerator` と同一ファイル内に **`internal sealed record`** として宣言されている。同一モジュール内の `Internal/BlockInfo.cs` に **既に同一の型定義が存在する**にも拘わらず、重複定義が残存している。

これはコンパイルエラーに至る可能性がある **型の二重定義** であり、名前空間の違い（`Observability` vs `Observability.Internal`）によって現時点では回避されているが、意図的な設計とは言えない。

**影響**: 保守性の低下。どちらの `BlockInfo` が使用されているか不明確となり、リファクタリング時の混乱を招く。

---

### ❌ ハッシュ計算ロジックの重複 (DRY 原則違反)

**場所**: [`PermissionsCatalogGenerator.cs`](/src/BuildingBlocks/Generators/Authorization/PermissionsCatalogGenerator.cs) L261-L283

`PermissionsCatalogGenerator` は `UpdateHash()` メソッドとして**独自のFNV-1a ハッシュ実装**を内包している。モジュール全体で共通利用される [`Fnv1aHash`](/src/BuildingBlocks/Generators/Utilities/Fnv1aHash.cs) ユーティリティが存在するにも拘わらず、これを使用していない。

他のジェネレーター（`AuthenticationMetadataGenerator`, `AuthorizationMetadataGenerator`, `ErrorCatalogGenerator`）は全て `Fnv1aHash.Compute()` を正しく使用しており、`PermissionsCatalogGenerator` のみが独自実装を持つ点は**一貫性を欠く**。

**影響**: ハッシュアルゴリズム変更時に `PermissionsCatalogGenerator` が取り残されるリスク。また、バグ修正が全箇所に適用されない危険性。

## 🛡️ 非機能要件とセキュリティ (Non-Functional Requirements & Security)

### 🔒 生成コードのアクセシビリティ

**低リスク**: 多くのジェネレーターは生成クラスを `internal static class` として出力しているが、以下のジェネレーターは `public` クラスを生成している：

| ジェネレーター | 生成クラス | アクセス修飾子 |
|---|---|---|
| `AuthorizationMetadataGenerator` | `AuthorizationMetadata` | `public static` |
| `PermissionsCatalogGenerator` | `PermissionsCatalog` | `public static` |
| `AuthorizationHandlersGenerator` | `GeneratedAuthorizationServiceCollectionExtensions` | `public static` |
| `EnumPolicyGenerator` | Requirement / Attribute / Handler | `public` |
| `PermissionHandlerGenerator` | PermissionProvider | `public` |

これらの `public` 宣言は、生成されたコードが**外部パッケージからも参照可能**であることを意味する。意図的設計であれば問題ないが、不必要な API サーフェスの拡大を避けるために `internal` へのダウングレードを検討すべき。

### 🔒 `VKBlockGeneratorGuard` の名前ベースマッチング

**場所**: [`VKBlockGeneratorGuard.cs`](/src/BuildingBlocks/Generators/Utilities/VKBlockGeneratorGuard.cs) L49

`assemblyName!.Contains(leafNamespace)` による文字列部分一致は、偶発的なマッチング（例：`ThirdParty.Authentication.Utils` が `Authentication` ジェネレーターをトリガーする）のリスクがある。`StartsWith($"VK.Blocks.{leafNamespace}")` への変更でマッチング精度を向上させるべき。

## 🧪 テスト容易性と疎結合性 (Testability & Decoupling)

### ⚙️ テストプロジェクトの不在

現時点で `VK.Blocks.Generators.Tests` プロジェクトが存在しない。Source Generator のテストは `Microsoft.CodeAnalysis.CSharp.CSharpGeneratorDriver` を使用して体系的に検証可能であり、最低限以下のテストが推奨される：

- 各ジェネレーターの Happy Path テスト（正しい属性付きクラスからの生成）
- `VKBlockGeneratorGuard.ShouldExecute` のユニットテスト
- ジェネレーター間の出力干渉テスト（例：Authentication と Authorization が同一アセンブリに存在する場合）

### ⚙️ 疎結合設計の評価

ジェネレーター間の **直接依存は皆無** であり、各ジェネレーターは完全に独立して動作する。これは **優れた疎結合性** と評価できる。

共通ユーティリティ（`SymbolExtensions`, `IdentifierUtilities`, `Fnv1aHash`, `VKBlockGeneratorGuard`）は、全て `internal static` クラスとして設計されており、Source Generator の制約（netstandard2.0, BI 分離）を正確に理解した上での設計判断と認められる。

## 🔭 可観測性の準拠度 (Observability Readiness)

### 📡 `ObservabilityAnalyzer` の網羅性

[`ObservabilityAnalyzer`](/src/BuildingBlocks/Generators/Observability/ObservabilityAnalyzer.cs) は、認可ハンドラーに対するメトリクス記録の欠如を検出する Roslyn Analyzer として、**Rule 6 の自動化執行を担う先進的な実装**である。

ただし、現在の検出対象は以下のパターンに限定されている：

- `IAuthorizationHandler` 実装クラス
- `[Diagnosable]` 属性付きクラス

MediatR Handler や Application Layer の Query/Command Handler への拡張を検討することで、可観測性の網羅性を向上できる。

### 📡 `VKBlockDiagnosticsGenerator` の自動化

[`VKBlockDiagnosticsGenerator`](/src/BuildingBlocks/Generators/Observability/VKBlockDiagnosticsGenerator.cs) は `[VKBlockDiagnostics]` 属性から `ActivitySource` と `Meter` を自動生成しており、**Rule 6 のインフラストラクチャ要件を完全に自動化**している。これは高い評価に値する。

### 📡 `ErrorCatalogGenerator` によるエラートポロジ

[`ErrorCatalogGenerator`](/src/BuildingBlocks/Generators/Diagnostics/ErrorCatalogGenerator.cs) は全モジュールの `Error` 定義をスキャンし、完全なカタログとメタデータハッシュを生成する。i18n サポートおよびランタイム診断に直接貢献する**戦略的実装**。

## ⚠️ コード品質とコーディング規約のリスク (Code Quality & Standard Risks)

### ⚠️ 生成コード内の `#region` ディレクティブ (規約違反)

**場所**: [`PermissionsCatalogGenerator.cs`](/src/BuildingBlocks/Generators/Authorization/PermissionsCatalogGenerator.cs) L235, L249

```csharp
sb.AppendLine("    #region Typed Permission Attributes");
// ...
sb.AppendLine("    #endregion");
```

VK.Blocks コーディング規約では `#region` の使用を禁止している。生成コードであっても同一の規約を適用すべきであり、`#region` を削除し、コメントで区切りを表現すべき。

---

### ⚠️ 生成ヘッダーの不統一

4パターンの生成ヘッダーが混在している：

| パターン | 使用ファイル |
|---|---|
| `SourceCodeBuilder.CreateWithHeader()` (推奨) | `AuthorizationMetadataGenerator`, `EnumPolicyGenerator`, `PermissionHandlerGenerator` |
| 手動 `sb.AppendLine("// <auto-generated/>");` | `AuthenticationMetadataGenerator`, `ClaimsProviderDiscoveryGenerator`, `OAuthProviderMapperGenerator`, `AuthorizationHandlersGenerator`, `MinimumRankGenerator`, `PermissionsCatalogGenerator`, `ErrorCatalogGenerator` |
| Raw string literal | `VKBlockDiagnosticsGenerator` |
| `#nullable enable` の有無が不統一 | 複数ファイル |

全てのジェネレーターを `SourceCodeBuilder.CreateWithHeader()` に統一し、一貫した `// <auto-generated/>` + `#nullable enable` ヘッダーを保証すべき。

---

### ⚠️ 名前空間スタイルの不統一

生成コードの一部はブロックスコープ名前空間（`namespace X.Y { ... }`）を使用し、他はファイルスコープ名前空間（`namespace X.Y;`）を使用している：

| スタイル | 使用ジェネレーター |
|---|---|
| ブロックスコープ | `AuthorizationMetadataGenerator`, `EnumPolicyGenerator`, `AuthorizationHandlersGenerator`, `MinimumRankGenerator`, `PermissionsCatalogGenerator`, `PermissionHandlerGenerator`, `ErrorCatalogGenerator` |
| ファイルスコープ | `AuthenticationMetadataGenerator`, `VKBlockDiagnosticsGenerator` |

Source Generator 出力コードでは netstandard2.0 制約を考慮してブロックスコープ名前空間に統一することを推奨する。ただし、消費側が C# 12+ であることを前提にファイルスコープに統一する選択肢もある。

---

### ⚠️ `ClaimsProviderDiscoveryGenerator` の `ProviderInfo` レコードの同一ファイル内宣言

**場所**: [`ClaimsProviderDiscoveryGenerator.cs`](/src/BuildingBlocks/Generators/Authentication/ClaimsProviderDiscoveryGenerator.cs) L111

```csharp
private sealed record ProviderInfo(string FullName);
```

Rule 14 では「1ファイル1型」を原則とするが、`private sealed record` は「同一クラス内でのみ使用される private ネスト型」例外に該当する。しかし、以下のジェネレーターでも同様のパターンが見られ、**Internal/ サブディレクトリへの抽出が一貫して行われていない**：

| ジェネレーター | 同居レコード | 抽出済み？ |
|---|---|---|
| `ClaimsProviderDiscoveryGenerator` | `ProviderInfo` | ❌ (private) |
| `AuthorizationHandlersGenerator` | `HandlerInfo` | ❌ (private) |
| `EnumPolicyGenerator` | `EnumPolicyInfo` | ❌ (private) |
| `PermissionHandlerGenerator` | `ProviderInfo` | ❌ (private) |

これらは全て `private sealed record` であるため Rule 14 の例外条項に該当するが、モジュール内での**一貫性**を考慮すると、`Internal/` へ統一抽出することが望ましい。

---

### ⚠️ `AuthorizationHandlersGenerator` の非ファイルスコープ名前空間

**場所**: [`AuthorizationHandlersGenerator.cs`](/src/BuildingBlocks/Generators/Authorization/AuthorizationHandlersGenerator.cs) L134-L161

生成コードで `namespace VK.Blocks.Authorization.Generated { }` ブロックを使用しているのに対し、ソースファイル自体はファイルスコープ名前空間 `namespace VK.Blocks.Generators.Authorization;` を使用している。この**二重標準は許容**されるが、ドキュメントで明示すべき。

---

### ⚠️ `PermissionsCatalogGenerator.GetPermissionFromUsage` の戻りバリエーション

**場所**: [`PermissionsCatalogGenerator.cs`](/src/BuildingBlocks/Generators/Authorization/PermissionsCatalogGenerator.cs) L73-L89

`foreach` ループ内で複数の引数を調べるが、**最初にマッチした定数値のみ**を返す。`[AuthorizePermission("read", "write")]` のような複数引数パターンに対応していない可能性がある。現行の使用パターンが単一引数のみであれば現時点では問題ないが、将来的な拡張性に懸念が残る。

## ✅ 評価ポイント (Highlights / Good Practices)

### ✅ Incremental Generator の一貫した採用

全てのジェネレーターが `IIncrementalGenerator` を実装しており、`ISourceGenerator`（レガシーAPI）は一切使用されていない。これにより**IDE パフォーマンスの最適化**が保証されている。

### ✅ `VKBlockGeneratorGuard` による中央集権的実行ガード

全ジェネレーターが `VKBlockGeneratorGuard.ShouldExecute()` を使用してアセンブリスコープの実行制御を行っている。これにより、無関係なアセンブリでの不必要なジェネレーター実行を防止し、**ビルドパフォーマンスの維持**に貢献している。

### ✅ メタデータハッシュによる変更検出

`AuthenticationMetadataGenerator`, `AuthorizationMetadataGenerator`, `PermissionsCatalogGenerator`, `ErrorCatalogGenerator` は全てFNV-1aハッシュを生成し、**トポロジ変更の検出メカニズム**を提供している。これは運用監視とキャッシュ無効化に直接寄与する先進的パターン。

### ✅ `OAuthProviderMapperGenerator` の重複検出

`_duplicateProviderWarning` ダイアグノスティックにより、同一プロバイダに対する複数マッパーをコンパイル時に検出できる。これは**開発者体験の向上**と**静的解析による品質保証**の好例。

### ✅ `ObservabilityAnalyzer` によるルール自動執行

VK.Blocks Rule 6 の要件を Roslyn Analyzer として自動化し、**コードレビューに依存しない品質ゲート**を実現している。

### ✅ フィーチャー駆動型フォルダ構成

`Authentication/`, `Authorization/`, `Diagnostics/`, `Observability/` という機能領域別の構成は Rule 14 に準拠しており、各ジェネレーターの責務範囲が明確。

### ✅ `Polyfills.cs` による netstandard2.0 互換性確保

`IsExternalInit` のポリフィルにより、netstandard2.0 をターゲットとしつつ C# の `record` 型や `init` プロパティが使用可能。適切な `[EditorBrowsable(Never)]` 属性により開発者への可視性も制御されている。

### ✅ `ForAttributeWithMetadataName` の活用

`OAuthProviderMapperGenerator` と `VKBlockDiagnosticsGenerator` は Roslyn 4.3+ の `ForAttributeWithMetadataName` API を使用し、`CreateSyntaxProvider` + 手動フィルタより**効率的な属性発見**を実現している。他のジェネレーターへの横展開を推奨。

## 💡 改善ロードマップ (Evolutionary Roadmap)

### 1. 最優先対応 (Immediate Action)

| # | 項目 | 対象ファイル | 深刻度 |
|---|---|---|---|
| 1 | `VKBlockDiagnosticsGenerator.cs` 内の `BlockInfo` 重複定義を削除し、`Internal/BlockInfo.cs` のみを使用するよう修正 | `Observability/VKBlockDiagnosticsGenerator.cs` | 🔴 高 |
| 2 | `PermissionsCatalogGenerator` の `UpdateHash()` を削除し、`Fnv1aHash.Compute()` に統一 | `Authorization/PermissionsCatalogGenerator.cs` | 🔴 高 |

### 2. リファクタリング提案 (Refactoring)

| # | 項目 | 詳細 |
|---|---|---|
| 1 | **生成ヘッダーの統一**: 全ジェネレーターを `SourceCodeBuilder.CreateWithHeader()` に統一 | 手動ヘッダー構築を排除し、`#nullable enable` の統一を保証 |
| 2 | **`VKBlockGeneratorGuard` のマッチング精度向上**: `Contains(leafNamespace)` → `StartsWith($"VK.Blocks.{leafNamespace}")` に変更 | サードパーティアセンブリの誤マッチを防止 |
| 3 | **`ForAttributeWithMetadataName` への移行**: `CreateSyntaxProvider` + `InheritsFromOrIs` / `ImplementsInterface` を使用するジェネレーターを、より効率的な `ForAttributeWithMetadataName` に移行検討 | `AuthenticationMetadataGenerator`, `AuthorizationMetadataGenerator` 等が対象 |
| 4 | **名前空間スタイルの統一**: 生成コード出力の名前空間スタイルを統一 | ブロックスコープまたはファイルスコープの **いずれか** に統一 |
| 5 | **`IdentifierUtilities.ToPascalCase` の堅牢化**: アンダースコア区切り文字列（`my_permission`）への対応追加 | 現在は先頭文字のみ大文字化しており、パスカルケース変換が不完全 |

> **監査結論**: Generators モジュールは、VK.Blocks フレームワークの中核インフラとして**堅実な設計基盤**を有している。主要な懸念はコードの一貫性（ヘッダー統一、ハッシュ重複、`#region` 除去）に集中しており、アーキテクチャ上の致命的欠陥は検出されなかった。最優先の3項目を解消することで、スコア 90+ への到達が見込まれる。
