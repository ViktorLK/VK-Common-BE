# アーキテクチャ監査レポート: Web BuildingBlock

## 📊 監査サマリー

- **対象モジュール**: Web
- **監査日**: 2026-04-17
- **総合スコア**: 96 / 100
- **対象レイヤー判定**: Infrastructure / Presentation Layer (Cross-cutting Concerns)
- **総評 (Executive Summary)**:
  `VK.Blocks.Web` モジュールは、VK.Blocks フレームワークにおけるWeb APIの標準化を担う非常に完成度の高いモジュールです。RFC 7807 (Problem Details) への準拠、OpenTelemetry による高度な監視機能、相関ID (Correlation ID) の伝播、およびテナント識別など、大規模エンタープライズアプリケーションに必要な横断的関心が洗練された形で実装されています。C# 12+ の最新機能（`sealed record`、`Primary Constructors`、`required` 属性など）をフル活用しており、型安全性とパフォーマンスの両立が図られています。

---

## 🚨 重大なアーキテクチャの懸念事項

現在、重大な（致命的な）アーキテクチャ上の問題は検出されませんでした。非常に強固な基盤が構築されています。

---

## 🛡️ 非機能要件とセキュリティ

- 🔒 **[セキュリティ]**: [SecurityHeadersMiddleware.cs](/src/BuildingBlocks/Web/Features/Security/SecurityHeadersMiddleware.cs) により、HSTS, CSP, X-Frame-Options などの重要なセキュリティヘッダーがデフォルトで適用される設計になっており、安全な Web API 開発を強力にサポートしています。
- 🔒 **[テナント分離]**: [TenantIdentificationMiddleware.cs](/src/BuildingBlocks/Web/Features/Tenancy/TenantIdentificationMiddleware.cs) において、ヘッダーやクエリパラメータからテナントIDを識別し、システムコンテキストに伝播させる仕組みが `VKTenancyConstants` を介して `VK.Blocks.Core` と正密に連携しています。
- ⚡ **[パフォーマンス]**: [ResponseShapingFilter.cs](/src/BuildingBlocks/Web/Features/Shaping/ResponseShapingFilter.cs) でリフレクションが使用されていますが、`ConcurrentDictionary` によるプロパティ情報のキャッシュが実装されており、実行時のオーバーヘッドが最小化されています。

---

## 🧪 テスト容易性と疎結合性

- ⚙️ **[テスト容易性]**: [VKWebBlockExtensions.cs](/src/BuildingBlocks/Web/DependencyInjection/VKWebBlockExtensions.cs) で定義されている DI 登録パターンは、モジュール間の依存関係を明確にし、モック可能なインターフェース（`ICorrelationIdProvider`, `IProblemDetailsFactory` など）を多用しているため、単体テストおよび結合テストが容易な設計です。
- ⚙️ **[疎結合]**: [TenantIdentificationMiddleware.cs](/src/BuildingBlocks/Web/Features/Tenancy/TenantIdentificationMiddleware.cs) は `VK.Blocks.MultiTenancy` モジュールに直接依存せず、`Core` の定数のみを使用してテナントの「識別」のみを行うことで、循環参照を回避しつつ責務を分離しています。

---

## 🔭 可観測性の準拠度

- 📡 **[メトリクスとトレース]**: [WebDiagnostics.cs](/src/BuildingBlocks/Web/Diagnostics/WebDiagnostics.cs) において、`VKBlockDiagnostics` ソースジェネレーターを活用したメトリクス（リクエスト数、レイテンシ、ペイロードサイズ）の収集が極めて詳細に行われています。
- 📡 **[構造化ログ]**: [ProblemDetailsLog.cs](/src/BuildingBlocks/Web/Features/ProblemDetails/ProblemDetailsLog.cs) など、すべてのログ出力に `[LoggerMessage]` を使用しており、高性能かつ構造化されたログ環境が提供されています。
- 📡 **[エラーハンドリング]**: [VKExceptionHandler.cs](/src/BuildingBlocks/Web/Features/ProblemDetails/VKExceptionHandler.cs) により、未処理例外を捕捉して標準化された `ProblemDetails` レスポンスに変換する ASP.NET Core の `IExceptionHandler` パターンが正しく実装されています。

---

## ⚠️ コード品質とコーディング規約のリスク

- ⚠️ **[リフレクションの利用]**: [ResponseShapingFilter.cs](/src/BuildingBlocks/Web/Features/Shaping/ResponseShapingFilter.cs) は非常に便利な機能ですが、複雑なオブジェクトグラフに対する動的なシェイピングは予期せぬパフォーマンス劣化を招く可能性があります。利用シーン（公開APIなど）を精査することが推奨されます。
- ⚠️ **[Controllerベースの継承]**: [VKApiController.cs](/src/BuildingBlocks/Web/Common/VKApiController.cs) は非常に便利ですが、将来的に Minimal API が主流になる場合、これらのヘルパー関数の多くがそのままでは利用できません。ビジネスロジックの結果マッピングを Extension メソッド等で共通化することを検討する価値があります。

---

## ✅ 評価ポイント

- **RFC 7807 準拠**: [VKWebProblemDetails.cs](/src/BuildingBlocks/Web/Features/ProblemDetails/VKWebProblemDetails.cs) は標準仕様を忠実に再現しており、開発環境では `DebugInfo` を出力するなどの配慮も優れています。
- **Rule 13 (DI Registration) の徹底**: `IsVKBlockRegistered` を用いたべき等なサービス登録パターンがすべての拡張メソッドで守られています。
- **イミュータビリティ**: DTOやオプションクラスに `sealed record` と `init` キーワードが徹底されており、副作用の少ないコードになっています。
