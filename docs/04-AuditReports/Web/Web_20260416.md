## 📊 監査サマリー (Audit Summary)

- **総合スコア**: **88 / 100**
- **対象レイヤー判定**: Presentation / Infrastructure Layer（ミドルウェアパイプライン、APIエンベロープ、例外ハンドリング）
- **総評 (Executive Summary)**:

Web ビルディングブロックは、VK.Blocks のアーキテクチャ標準に対して高い準拠度を示しています。`[VKBlockDiagnostics]` Source Generator によるテレメトリ一元化、`[LoggerMessage]` SG ベースの構造化ログ、`IVKBlockOptions` による Zero-Reflection 構成解決、および Rule 13 準拠の冪等サービス登録パターンが全面的に実装されています。

しかし、いくつかの構造的改善点が残存しています。`Internal/` フォルダに残る汎用名のキャッシュクラス、`Diagnostics/Controllers/` の空ディレクトリ、`VKApiController` のジェネリック `PagedResult<object>` パターンマッチング問題、`RequestLoggingMiddleware` のマジックストリング、および `VKWebProblemDetails` が `sealed record` ではなく `sealed class` である非対称性が主な指摘事項です。

## 🚨 重大なアーキテクチャの懸念事項 (Critical Architectural Smells)

- ❌ **ジェネリック型パターンマッチング不整合**: [[VKApiController.cs](/src/BuildingBlocks/Web/Common/VKApiController.cs:L202)] - `HandleResult<T>` 内で `PagedResult<object>` に対してパターンマッチングを行っているが、実際の呼び出し元は `PagedResult<SomeDto>` 型を返すため、ジェネリック型の共変性により **このブランチに到達しない可能性が極めて高い**。`PagedResult<CustomerDto>` は `PagedResult<object>` ではないため、常に `ApiResponse<T>.SuccessResult` ブランチに到達する。これは **機能的なバグ** であり、ページネーションレスポンスが正しくラップされない可能性がある。

```csharp
// 現行（到達不能コード）
result.Value switch
{
    PagedResult<object> pagedResult => Ok(PagedResponse<object>.SuccessResult(pagedResult)),
    _ => Ok(ApiResponse<T>.SuccessResult(result.Value))
};
```

**影響度**: 高 — ページ付きAPIレスポンスが `PagedResponse<T>` ではなく `ApiResponse<PagedResult<T>>` として返却され、フロントエンドのパース仕様と不一致を引き起こすリスクがある。

- ❌ **マジックストリング参照**: [[RequestLoggingMiddleware.cs](/src/BuildingBlocks/Web/Features/RequestLogging/RequestLoggingMiddleware.cs:L47)] - `context.Items["CorrelationId"]` というハードコード文字列で Correlation ID を取得しているが、`WebConstants.Items.CorrelationId` 定数が既に定義されている。定数を使用すべきである。

```csharp
// 現行（マジックストリング）
var correlationId = context.Items["CorrelationId"]?.ToString();
// 修正案
var correlationId = context.Items[WebConstants.Items.CorrelationId]?.ToString();
```

## 🛡️ 非機能要件とセキュリティ (Non-Functional Requirements & Security)

- 🔒 **`SecurityDiscoveryController` の認可チェック欠如**: [[SecurityDiscoveryController.cs](/src/BuildingBlocks/Web/Features/Discovery/SecurityDiscoveryController.cs:L56)] - セキュリティトポロジーを公開するエンドポイントであるにもかかわらず、`[Authorize]` 属性が設定されていない。`SecurityDiscoveryOptions.Enabled = false` がデフォルトであり、ガードとして `GuardEnabled()` が実装されているが、設定ミスにより有効化された場合、**認証なしでセキュリティメタデータが漏洩するリスク** がある。最低限 `[Authorize(Roles = "Admin")]` または管理者ポリシーの適用を推奨する。

- 🔒 **`VKWebProblemDetails.DebugInfo` の本番環境漏洩リスク**: [[VKWebProblemDetails.cs](/src/BuildingBlocks/Web/Features/ProblemDetails/VKWebProblemDetails.cs:L45)] - `DebugInfo` プロパティに `[JsonIgnore(Condition = JsonIgnoreCondition.WhenWritingNull)]` が適用されているが、`VKExceptionProblemDetailsMapper` が `DebugInfo` を常にマッピングするため、本番環境での `IHostEnvironment.IsDevelopment()` チェックがマッパー側に存在しない。`VKWebDebugInfo` にはスタックトレースや例外型名が含まれ、**情報開示攻撃 (CWE-209)** のリスクがある。

- 🔒 **`SecurityHeadersMiddleware` で Strict-Transport-Security (HSTS) ヘッダーが未設定**: [[SecurityHeadersMiddleware.cs](/src/BuildingBlocks/Web/Features/Security/SecurityHeadersMiddleware.cs)] - OWASP 推奨の主要セキュリティヘッダーのうち `Strict-Transport-Security` (HSTS) と `Permissions-Policy` が `SecurityHeadersOptions` に含まれていない。ASP.NET Core の UseHsts() との棲み分けを明確にした上で、オプショナルプロパティとして追加することを推奨する。

- 🔒 **`TenantIdentificationMiddleware` のテナント入力バリデーション不足**: [[TenantIdentificationMiddleware.cs](/src/BuildingBlocks/Web/Features/Tenancy/TenantIdentificationMiddleware.cs:L96-L115)] - ヘッダーまたはクエリストリングから受け取った `tenantId` を **サニタイズなしで `HttpContext.Items` に格納** している。テナントIDが GUID 形式であるべき場合、フォーマットバリデーション（`Guid.TryParse` 等）を追加することで、ログインジェクション攻撃や不正なテナントID混入を防止できる。

## 🧪 テスト容易性と疎結合性 (Testability & Decoupling)

- ⚙️ **`IProblemDetailsFactory` の実装クラス未提供**: [[IProblemDetailsFactory.cs](/src/BuildingBlocks/Web/Features/ProblemDetails/Abstractions/IProblemDetailsFactory.cs)] - インターフェースが定義されているが、`VKExceptionHandler` のコンストラクタで DI 注入されているにもかかわらず、**Web ブロック内にデフォルト実装が存在しない**。`VKWebBlockExtensions.AddVKWebBlock()` 内でも登録されていない。これは外部ブロックからの登録に依存していることを意味するが、依存関係が暗黙的であり、登録漏れ時にランタイムの `InvalidOperationException` を誘発する。

- ⚙️ **`VKApiController` が `abstract class` であり `sealed` ではない**: [[VKApiController.cs](/src/BuildingBlocks/Web/Common/VKApiController.cs:L68)] - Rule 12 では `sealed class` がデフォルトであるが、基底コントローラーとして継承が意図されているため `abstract` は適切。ただし、`HandleResult` メソッドが `protected` かつ `virtual` でないため、サブクラスでのオーバーライドが不可能であり、テスト時のモック戦略に制約がある点は認識すべき。

- ⚙️ **`WebDiagnosticsMiddleware` が `IUserContext` に直接依存**: [[WebDiagnosticsMiddleware.cs](/src/BuildingBlocks/Web/Diagnostics/WebDiagnosticsMiddleware.cs:L33)] - ミドルウェアパイプラインの最外周で `IUserContext` を注入しているが、テナント情報が確定する前（`TenantIdentificationMiddleware` 実行前）に呼び出される可能性がある。`finally` ブロック内で `context.Items[WebConstants.Items.TenantId]` へのフォールバックが実装されているため実害はないが、DI の意図が不明瞭である。

## 🔭 可観測性の準拠度 (Observability Readiness)

- 📡 **ロギング標準準拠**: ✅ 全ミドルウェアおよび例外ハンドラーで `[LoggerMessage]` Source Generator ベースの構造化ロギングが使用されている。`CorrelationIdLog`, `TenancyLog`, `RequestLoggingLog`, `ProblemDetailsLog` が各フィーチャーフォルダ内に適切に配置されている。

- 📡 **メトリクス計装**: ✅ `WebDiagnostics` クラスが `[VKBlockDiagnostics]` アトリビュートにより `ActivitySource` と `Meter` を自動生成し、7つのメトリクスインストルメント（`Counter`, `UpDownCounter`, `Histogram`）を集中管理している。定数は `WebDiagnosticsConstants` に完全に分離されている。

- 📡 **分散トレーシング**: ✅ 全ミドルウェアが `WebDiagnostics.Source.StartActivity()` を使用してスパンを生成しており、OpenTelemetry に準拠したタグ付けが行われている。

- 📡 **軽微な非準拠**: `RequestLoggingLog.LogRequestStarted` で `CorrelationId` パラメータが `string?` (nullable) であり、構造化テンプレート内で null が出力される可能性がある。ログパイプラインの下流で `"null"` 文字列として出力されるリスクは低いが、`?? "none"` のデフォルト値を検討すべきである。

## ⚠️ コード品質とコーディング規約のリスク (Code Quality & Standard Risks)

- ⚠️ **`VKWebProblemDetails` が `sealed class` であり `sealed record` ではない**: [[VKWebProblemDetails.cs](/src/BuildingBlocks/Web/Features/ProblemDetails/VKWebProblemDetails.cs)] - Rule 12 では DTO に `sealed record` を推奨しているが、`Microsoft.AspNetCore.Mvc.ProblemDetails` を継承しているため `record` にはできない（基底クラスが `class`）。これは技術的制約であり正当だが、XML コメントでその理由を明示すべきである。現行のコメントは `sealed` の理由は述べているが `record` でない理由は未記載。

- ⚠️ **`Internal/` フォルダの汎用命名**: [[ValidationFailureCache.cs](/src/BuildingBlocks/Web/Internal/ValidationFailureCache.cs)] - Rule 14 に基づき、`Internal/` は概念ベースのフォルダに置き換えるべきである。`ValidationFailureCache` はバリデーション結果のキャッシュであるため、`Features/Validation/` または `Caching/` へ移動することが望ましい。

- ⚠️ **空ディレクトリ `Diagnostics/Controllers/`**: [[Diagnostics/Controllers/](/src/BuildingBlocks/Web/Diagnostics/Controllers/)] - 空のディレクトリが残存している。リファクタリング途中の残骸と思われるが、バージョン管理上の不整合を避けるため削除すべきである。

- ⚠️ **`RegisterSecurityDiscoveryServices` の副作用**: [[VKWebBlockExtensions.cs](/src/BuildingBlocks/Web/DependencyInjection/VKWebBlockExtensions.cs:L123-L127)] - `services.AddControllers().AddApplicationPart(...).AddControllersAsServices()` は **冪等ではなく、他のブロックの MVC 構成に影響** する。`AddControllersAsServices()` は全コントローラーの DI ライフサイクルを変更するため、意図しない副作用が発生する可能性がある。Feature フラグ付きコントローラーパターンへの移行を検討すべきである。

- ⚠️ **`WithTenantIdentification` メソッドが空実装**: [[VKWebBlockExtensions.cs](/src/BuildingBlocks/Web/DependencyInjection/VKWebBlockExtensions.cs:L155-L159)] - Builder パターンのメソッドが空であり「整合性のため」とのコメントがあるが、ミドルウェアのオプションやフォーマットバリデーション機能が追加される可能性があるなら、`TenantIdentificationOptions` の導入を検討すべきである。現行のまま空メソッドを公開 API として維持する場合は YAGNI 原則に基づき削除も選択肢である。

- ⚠️ **`TODO` コメントの残存**: [[VKWebBlockExtensions.cs](/src/BuildingBlocks/Web/DependencyInjection/VKWebBlockExtensions.cs:L61)] - `// TODO: [Mapping SG] Remove this manual registration once Source Generator auto-registration is implemented.` が残存。Rule 10 では TODO を禁止しているが、これはコード生成ではなく既存コードの注釈であるため、技術的債務の追跡として許容する。ただし、Issue トラッカーに紐づけるべきである。

- ⚠️ **`CorrelationIdMiddleware.Invoke` メソッド名の不統一**: [[CorrelationIdMiddleware.cs](/src/BuildingBlocks/Web/CorrelationId/CorrelationIdMiddleware.cs:L44)] - `Invoke` を使用しているが、`SecurityHeadersMiddleware` では `InvokeAsync` を使用している。ASP.NET Core のミドルウェア規約ではどちらも許容されるが、**プロジェクト内での統一性** が欠如している。`InvokeAsync` に統一することを推奨する。

## ✅ 評価ポイント (Highlights / Good Practices)

- ✅ **Rule 13 完全準拠**: `VKWebBlock` マーカー型、`IsVKBlockRegistered<VKWebBlock>()` による冪等チェック、`AddVKBlockMarker<VKWebBlock>()` による成功コミット、`CoreBlock` の前提条件チェックが全て実装されている。
- ✅ **`IVKBlockOptions` Zero-Reflection パターン**: `CorrelationIdOptions`, `RequestLoggingOptions`, `SecurityHeadersOptions`, `SecurityDiscoveryOptions`, `VKCorsOptions` の全 5 Options クラスが `IVKBlockOptions` を実装し、`SectionName` を `VKBlocks:Web:*` 階層で定義している。
- ✅ **`[VKBlockDiagnostics]` SG ベースの集中テレメトリ**: `WebDiagnostics` が SG により `ActivitySource` と `Meter` を自動生成し、7つのメトリクスを管理。`WebDiagnosticsConstants` が全メトリクス名とタグキーを定数化している。
- ✅ **Feature-Driven (Vertical Slice) 構造**: `Features/Cors`, `Features/Discovery`, `Features/ProblemDetails`, `Features/RequestLogging`, `Features/Security`, `Features/Tenancy`, `Features/UserContext` として明確にドメイン駆動型フォルダ構成が適用されている。
- ✅ **`[LoggerMessage]` SG 完全準拠**: 全フィーチャーフォルダに専用ログクラスが配置され、直接的な `logger.LogInformation()` 呼び出しが排除されている。
- ✅ **`ConfigureAwait(false)` 全量適用**: 全ミドルウェアの `await` 呼び出しで `.ConfigureAwait(false)` が適用されており、ライブラリコードとしての同期コンテキストデッドロック防止が徹底されている。
- ✅ **RFC 7807 準拠**: `VKWebProblemDetails` が `Microsoft.AspNetCore.Mvc.ProblemDetails` を拡張し、`ErrorCode`, `TraceId`, `Timestamp`, `DebugInfo`, `Errors` を追加したフルスタック RFC 7807 準拠を実現。
- ✅ **Builder パターン**: `IVKBlockBuilder<VKWebBlock>` のチェイナブルな `.With*()` メソッドにより、機能の選択的有効化が可能。各 `With*()` メソッドは Configuration オーバーロードと Action オーバーロードを提供している。
- ✅ **sealed by default**: `CorrelationIdMiddleware`, `RequestLoggingMiddleware`, `SecurityHeadersMiddleware`, `TenantIdentificationMiddleware`, `VKExceptionHandler`, `HttpContextUserContext` 等、全コンクリートクラスが `sealed` 宣言されている。
- ✅ **不変レコード DTO**: `ApiResponse<T>`, `PagedResponse<T>`, `VKWebDebugInfo`, `VKWebErrorDetail`, `VKCorsOptions`, `CorrelationIdOptions`, `RequestLoggingOptions`, `SecurityHeadersOptions`, `SecurityDiscoveryOptions` がすべて `sealed record` として定義されている。
- ✅ **`TryAddScoped` / `TryAddSingleton` パターン**: `VKWebBlockExtensions.AddVKWebBlock()` 内で `TryAddScoped<IUserContext, HttpContextUserContext>()` および `TryAddSingleton<IMapper<...>, VKExceptionProblemDetailsMapper>()` が使用されている。

### 3. 推奨される学習トピック (Learning Suggestions)

- **ジェネリック型の共変性と型パターンマッチング**: `PagedResult<T>` の正しいハンドリングには、C# のジェネリック共変性 (`out T`) 制約または非ジェネリックインターフェース (`IPagedResult`) の導入が有効。
- **ASP.NET Core ミドルウェアの実行順序設計**: 特に Diagnostics → Security → Context → Logging の順序における、各ミドルウェアの前提条件と DI スコープの関係の深い理解。
- **RFC 7807+ 拡張とエラートラクサビリティ**: `ProblemDetails` のカスタム拡張フィールドの設計パターンと、Production vs Development 環境での情報量の制御パターン。
