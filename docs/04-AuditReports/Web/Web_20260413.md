> **対象モジュール**: `src/BuildingBlocks/Web`
> **監査日**: 2026-04-13
> **アーキテクト**: VK.Blocks Lead Architect (Strict Mode)

## 📊 監査サマリー (Audit Summary)

- **総合スコア**: **82 / 100**
- **対象レイヤー判定**: Presentation / Infrastructure Layer (ASP.NET Core Web 基盤ブロック)
- **総評 (Executive Summary)**:

Web BuildingBlock は、ASP.NET Core アプリケーションにおける HTTP リクエスト/レスポンスの標準化を担うインフラストラクチャレイヤーのモジュールである。`Result<T>` → HTTP レスポンスのマッピング、RFC 7807 準拠の `VKProblemDetails`、統一レスポンスエンベロープ (`ApiResponse<T>`, `PagedResponse<T>`)、Correlation ID の伝播、そして `IUserContext` のHTTPコンテキストベース実装という、Web API 基盤に必要な機能を適切に提供している。

Rule 13 のサービス登録パターン (`WebBlockExtensions`) の実装は模範的であり、Check-Self → Check-Prerequisite → Actual Registration → Mark-Self の4ステップが忠実に遵守されている。しかし、**CorrelationId の登録メソッドがこの統一パターンに統合されていない**という構造的な不整合が存在する。また、ライブラリコードにおける `ConfigureAwait(false)` の欠落 (Rule 3) や、一部の依存クラスにおける `sealed` 修飾子の欠落 (Rule 12) など、ゼロトレランスルールへの違反が散見される。

全体として堅実な基盤であるが、フレームワークとしての完全性を担保するために数点の改善が必要である。

## 🚨 重大なアーキテクチャの懸念事項 (Critical Architectural Smells)

### ❌ Rule 13 違反 — CorrelationId 登録における直接 `AddScoped` 使用

**該当箇所**: [ServiceCollectionExtensions.cs:35](/src/BuildingBlocks/Web/Extensions/ServiceCollectionExtensions.cs#L35), [ServiceCollectionExtensions.cs:43](/src/BuildingBlocks/Web/Extensions/ServiceCollectionExtensions.cs#L43)

```csharp
services.AddScoped<ICorrelationIdProvider, DefaultCorrelationIdProvider>();
```

Rule 13 では、BuildingBlock 内のすべてのサービス登録に `TryAdd` パターンの使用を義務付けている。`AddScoped` は冪等ではなく、複数回呼び出された場合にサービスの重複登録を引き起こす。さらに、`AddCorrelationId` メソッド自体が `WebBlockExtensions.AddVKWebBlock()` の4ステップパターンから独立しており、登録の一貫性が損なわれている。

**影響**: サービスコンテナの重複登録、予測不能なサービス解決、モジュール登録の一貫性の崩壊。

**推奨対応**:
```csharp
services.TryAddScoped<ICorrelationIdProvider, DefaultCorrelationIdProvider>();
```
また、CorrelationId の登録を `AddVKWebBlock()` のフルエンシーメソッド（`.WithCorrelationId()`）として統合すべきである。

### ❌ Rule 3 違反 — `ConfigureAwait(false)` の欠落

**該当箇所**: [CorrelationIdMiddleware.cs:38](/src/BuildingBlocks/Web/CorrelationId/CorrelationIdMiddleware.cs#L38)

```csharp
await next(context);  // ConfigureAwait(false) が欠落
```

Rule 3 では、BuildingBlock/ライブラリコード内のすべての `await` 呼び出しに `.ConfigureAwait(false)` の使用を義務付けている。ASP.NET Core ミドルウェアはライブラリコードとして位置付けられるため、同期コンテキストのデッドロック防止のために `ConfigureAwait(false)` が必須である。

> **補足**: ASP.NET Core はデフォルトで `SynchronizationContext` を持たないため、実質的な影響は限定的だが、ライブラリとしての規約遵守は「Zero Tolerance」ルールに該当するため、修正は必須である。

---

### ❌ Rule 12 違反 — `PagedResult<T>` の `sealed` 欠落 (Core モジュール側)

**該当箇所**: [Core/Results/PagedResult.cs:7](/src/BuildingBlocks/Core/Results/PagedResult.cs#L7)

```csharp
public class PagedResult<T>  // sealed キーワードが欠落
```

Web モジュールが依存する `PagedResult<T>` が `sealed` 宣言されていない。Rule 12 では、ポリモーフィズムが明示的に必要でない限り、すべてのクラスを `sealed` にすることを義務付けている。`PagedResult<T>` はデータ転送オブジェクトであり、継承の要件は考えにくい。

> **注**: これは Web モジュール自体ではなく Core モジュールの問題だが、依存関係の完全性として記録する。

## 🛡️ 非機能要件とセキュリティ (Non-Functional Requirements & Security)

### 🔒 パフォーマンス — `HttpContextUserContext.Roles` のアロケーション

**該当箇所**: [HttpContextUserContext.cs:57-62](/src/BuildingBlocks/Web/Context/HttpContextUserContext.cs#L57-L62)

```csharp
return user.FindAll(VKClaimTypes.Role)
    .Concat(user.FindAll(ClaimTypes.Role))
    .Select(c => c.Value)
    .Distinct()
    .ToList();
```

`Roles` プロパティはアクセスのたびに `FindAll` × 2、`Concat`、`Select`、`Distinct`、`ToList` を実行し、新しい `List<string>` を毎回ヒープにアロケートする。リクエストスコープ内で複数回アクセスされるケースでは、不要なアロケーションが蓄積する。

**推奨対応**: リクエストスコープ内でのキャッシュ（`Lazy<IReadOnlyList<string>>` や遅延初期化フィールド）を検討する。

**該当箇所**: [ValidationFailureCache.cs:14](/src/BuildingBlocks/Web/Caches/ValidationFailureCache.cs#L14)

```csharp
// "CRITICAL EORROR" → "CRITICAL ERROR" (タイポ)
```

## 🧪 テスト容易性と疎結合性 (Testability & Decoupling)

### ⚙️ 良好 — インターフェース抽象化の適切な使用

- `ICorrelationIdProvider` → `DefaultCorrelationIdProvider`: Strategy パターンによる差し替え可能な設計
- `IUserContext` → `HttpContextUserContext`: Core 層のインターフェースに対する Web 固有の実装
- すべての依存関係がコンストラクタインジェクションで注入されており、テスト時のモック化が容易

### ⚙️ 懸念 — `VKApiController` の直接的な `Activity.Current` 参照

**該当箇所**: [VKApiController.cs:28,44,60,106](/src/BuildingBlocks/Web/Presentation/VKApiController.cs#L28)

```csharp
Activity.Current?.Id ?? HttpContext.TraceIdentifier
```

`Activity.Current` は静的プロパティ (ambient context) であり、テスト時のインジェクション・モック化が困難である。このパターンはコントローラ内の4箇所で繰り返されており、DRY 原則にも違反している。

**推奨対応**: トレース ID 取得を専用のヘルパーメソッドやプロパティに抽出する。

```csharp
protected string CurrentTraceId => Activity.Current?.Id ?? HttpContext.TraceIdentifier;
```

### ⚙️ 懸念 — `ServiceCollectionExtensions` の責務過多

**該当箇所**: [ServiceCollectionExtensions.cs](/src/BuildingBlocks/Web/Extensions/ServiceCollectionExtensions.cs)

## 🔭 可観測性の準拠度 (Observability Readiness)

### 📡 良好 — Correlation ID の自動伝播

`CorrelationIdMiddleware` は `ILogger.BeginScope` を使用して Correlation ID をログスコープに自動的にプッシュしている。これにより、構造化ログプロバイダ (Serilog 等) との統合が保証される。また、OpenTelemetry の `Activity.TraceId` をフォールバックとして使用するフローも適切である。

### 📡 良好 — RFC 7807 準拠のエラーレスポンス

`VKProblemDetails` は `ProblemDetails` を継承し、`ErrorCode` と `TraceId` と `Timestamp` を拡張フィールドとして追加している。`ErrorTypeExtensions.ToStatusCode()` によるマッピングは網羅的であり、すべての `ErrorType` 列挙値をカバーしている。

### 📡 警告 — Rule 6 診断インフラの未実装

Web モジュールに `[VKBlockDiagnostics]` による `ActivitySource` / `Meter` の定義がない。ミドルウェアの処理時間やリクエスト数のメトリクス計測が未実装であり、Rule 6 (Observability — Metrics & Tracing) への準拠が不完全である。

### 📡 警告 — `[LoggerMessage]` Source Generator 未使用

`CorrelationIdMiddleware` では `ILogger` がコンストラクタパラメータに存在するが、実際にはログ出力が行われていない。本番運用では、CorrelationId の割り当てや伝播をログに記録すべきであり、その際は Rule 6 に従い `[LoggerMessage]` Source Generator を使用する必要がある。

## ⚠️ コード品質とコーディング規約のリスク (Code Quality & Standard Risks)

### ⚠️ `CorrelationIdOptions` — `required` キーワードの欠落

**該当箇所**: [CorrelationIdOptions.cs:4](/src/BuildingBlocks/Web/CorrelationId/CorrelationIdOptions.cs#L4)

Rule 12 では、DTO やドメイン設定には `sealed record` を推奨している。ただし、Options パターン (`IOptions<T>`) は `class` を前提とした設定バインディングフレームワークであるため、`sealed class` は許容範囲内。

> **補足**: Options パターンでは、`IConfiguration` からのバインディングが `set` アクセサを要求するため、`required` + `init` の適用は Options 型には不適切である。ここでは `IValidateOptions<T>` によるランタイムバリデーションが最適解。

---

### ⚠️ `AddApiStandards` の暗黙的な `AddVKWebBlock` 呼び出し

**該当箇所**: [ServiceCollectionExtensions.cs:26](/src/BuildingBlocks/Web/Extensions/ServiceCollectionExtensions.cs#L26)

```csharp
public static IServiceCollection AddApiStandards(this IServiceCollection services)
{
    services.AddProblemDetails();
    services.AddVKWebBlock();  // WebBlock の登録を暗黙的に委譲
    return services;
}
```

`AddApiStandards()` が内部的に `AddVKWebBlock()` を呼び出すが、戻り値型が `IServiceCollection` であり `IVKBlockBuilder<WebBlock>` ではない。呼び出し元が `AddVKWebBlock()` と `AddApiStandards()` の両方を呼んだ場合、`IsVKBlockRegistered` による冪等性で問題は起きないが、API の意図の明確性が低下している。

### ⚠️ `VKProblemDetails` の `required` キーワード欠落

**該当箇所**: [ExceptionHandling/Abstractions/Contracts/VKProblemDetails.cs:11](/src/BuildingBlocks/ExceptionHandling/Abstractions/Contracts/VKProblemDetails.cs#L11)

`VKProblemDetails` は `ProblemDetails` を継承しているため `record` への変更は不可能だが、`required` キーワードの欠落が指摘事項として残る（セッター経由でのプロパティ初期化漏れのリスク）。

> **注**: これは ExceptionHandling モジュール側の問題だが、Web モジュールの `VKApiController` がこれに直接依存しているため記録する。
