# ADR 012: Automated AI Evaluation Framework

- **Date**: 2026-05-10
- **Status**: 📝 Draft
- **Deciders**: Architecture Team
- **Technical Story**: VK.Blocks.AI / QA & DevOps

## 1. Context (背景)

LLM の出力は本質的に非決定的（Non-deterministic）です。コード修正やプロンプトの微調整、あるいはプロバイダー側のモデルアップデートによって、回答の質が密かに低下する「回帰（Regression）」を、従来のユニットテストだけで検知することは困難です。

## 2. Problem Statement (問題定義)

1. **品質の不透明性**: 「なんとなく良くなった気がする」という主観的な評価に頼らざるを得ない。
2. **大規模テストの困難さ**: 数百件のプロンプトを人間が手動でチェックするのは、コストと時間の面で現実的ではない。
3. **継続的改善の阻害**: 変更が品質を下げていないという確証がないため、積極的なリファクタリングや最適化を躊躇してしまう。

## 3. Decision (決定事項)

1. **`IVKAIEvaluator` インターフェースの定義**:
   AI の回答をスコアリングするための統一的な仕組みを定義します。
   - 入力：プロンプト、生成された回答、期待される回答（Ground Truth - オプション）。
   - 出力：0.0 〜 1.0 のスコアと、その根拠（Reasoning）。

2. **「LLM-as-a-Judge」パターンの採用**:
   より高性能なモデル（例：GPT-4o）を使用して、ターゲットモデル（例：Llama-3 や GPT-3.5）の回答を多角的（正確性、丁寧さ、安全性など）に評価する実装を標準提供します。

3. **CI/CD パイプラインへの統合**:
   プルリクエストごとに自動的に評価スクリプトを実行し、ベースライン（Baseline）スコアを下回った場合にビルドを失敗させる「Quality Gate」として機能させます。

4. **`VKAIQualityDashboard` の構築**:
   評価結果を時系列で保存し、品質の推移を可視化できるようにします。

## 4. Alternatives Considered (代替案の検討)

- **Option 1: 古典的な NLP 指標（BLEU, ROUGE）のみを使う**
  - **Rejected Reason**: これらの指標は単語の一致率を見るだけであり、意味的な正しさや文脈の妥当性を評価できないため。
- **Option 2: 人間による完全な手動評価**
  - **Rejected Reason**: スケーラビリティがなく、CI/CD サイクルを回すことができない。

## 5. Consequences & Mitigation (結果と緩和策)

- **Positive**: 
  - 品質の変化を「数値（KPI）」で客観的に証明できる。
  - プロンプトの A/B テストにおいて、データに基づいた決定ができる。
- **Negative**:
  - 評価自体にも LLM コストがかかる。
- **Mitigation**:
  - 評価をバッチ処理でまとめて行う、あるいは重要度の高いプロンプトのみに限定して評価を実行する構成を可能にします。

## 6. Implementation & Security (実装詳細とセキュリティ考察)

- **実装詳細**: 評価ロジックは `AI.Labs` または `AI.TestKit` モジュールとして提供し、本番のバイナリには含まれないように設計します。
- **セキュリティ考察**: 評価に使用する Ground Truth データには機密情報が含まれる可能性があるため、テスト用環境のデータ保護も本番同様に厳格に行う必要があります。
